# IN-GPS 온도 차트 최적화 작업 (2026-04-26)

---

## 작업 배경

모바일 앱에서 온도 로그 1년치 데이터를 조회할 때, 기존 방식은 서버에서 최대 10,000건의 raw 데이터를 전부 전송하고 Android 클라이언트에서 날짜 필터링을 수행했다. 이로 인해 화면 로드가 느리고 메모리 낭비가 발생하는 문제가 있었다.

---

## 해결 방안

> **데이터가 7일을 초과하면 일별 집계(평균)로 압축해서 전송, 최대 365개 포인트만 모바일로 전달**

- 7일 이하 조회 → raw 데이터 그대로 전송 (분 단위 상세)
- 7일 초과 조회 → 서버에서 `GROUP BY DATE` 일별 집계 후 전송 (하루 1개 평균값)

---

## 변경 내역

### 1. 서버 — 신규 엔드포인트 추가 (`app.py`)

**`GET /temperature/chart?device_id={id}&days={N}`**

| `days` | 반환 데이터 | 최대 건수 |
|--------|------------|----------|
| ≤ 7    | raw 데이터 (분 단위) | 수천 건 |
| > 7    | 일별 집계 (avg / min / max) | 최대 365건 |

집계 시 하루 중 가장 심각한 이벤트 우선 반영 (`disconnected > warning > normal`)

```
기존: GET /temperature?device_id=xxx&limit=10000
변경: GET /temperature/chart?device_id=xxx&days=N
```

---

### 2. 서버 — 더미 데이터 추가 (`temperature_db_init.sql`)

각 조회 구간을 테스트할 수 있도록 총 159건의 더미 데이터 구성

| 구간 | 기간 | 건수 | 목적 |
|------|------|------|------|
| 상세 raw | 2026-03-26 (2초 간격) | 30건 | 기존 데이터 유지 |
| 1년 뷰 테스트 | 2025-05 ~ 2026-02 (격주/주간) | 26건 | 계절별 온도 추이 + 여름 경고 이벤트 |
| 1개월 뷰 테스트 | 2026-03-01 ~ 03-25 (1건/일) | 25건 | 3월 말 경고 이벤트 포함 |
| 집계-raw 경계 | 2026-03-27 ~ 04-18 (2건/일) | 46건 | 4월 10~11일 고온 경고 이벤트 |
| 최근 7일 raw | 2026-04-19 ~ 04-26 (4건/일) | 32건 | 4월 22~24일 경고 이벤트 포함 |

### 5. Android — 차트 버그 수정 (`SensorDetailFragment.java`)

발견된 버그 4개 모두 수정

#### 버그 1 — `ORDER BY` 방향 불일치
- 서버가 `ASC`로 반환하는데 클라이언트가 `DESC`로 들어온다고 가정하고 `reverse()` 적용
- **수정**: 서버 `ORDER BY DESC`로 변경 → 클라이언트 `reverse()` 정상 동작

#### 버그 2 — 집계 데이터 날짜 포맷 불일치
- 서버 집계 `created_at`: `"2026-04-26 12:00:00"` (공백)
- Android `DATE_FORMAT`: `"yyyy-MM-dd'T'HH:mm:ss"` (T 기대)
- 파싱 실패 → `parseDate()`가 `new Date()` 반환 → x축 레이블 전부 `0h00m`으로 뭉개짐
- **수정**: `CONCAT(DATE(created_at), 'T12:00:00')` — T 구분자 사용

#### 버그 3 — MPAndroidChart viewport 미리셋
- 1년 뷰(83포인트) 후 zoom 스케일이 유지된 채로 1일 데이터(5포인트) 표시 시도
- `moveViewToX(5)` → x=5~25 범위 표시, 실제 데이터(x=0~4)가 viewport 밖으로 밀림
- **수정**: `chart.clear()` + `chart.fitScreen()` 으로 viewport 완전 리셋, `moveViewToX(max(0, pointIndex - 20))` 으로 마지막 포인트가 오른쪽 끝에 오도록 수정

#### 버그 4 — x축 레이블 가독성
- 기존: 첫 데이터 기준 경과 시간 (`0h00m`, `6h`, `341d`)
- **수정**: 실제 날짜/시간 표시
  - 조회 범위 2일 이내 → `HH:mm` (예: `00:00`, `06:00`, `18:00`)
  - 조회 범위 2일 초과 → `M/dd` (예: `4/22`, `4/25`)
